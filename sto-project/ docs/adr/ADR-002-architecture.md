# ADR-002: Вибір архітектурного стилю (Модульний моноліт)

## Статус: Accepted

## Контекст:
Команда розробки складається з 3 осіб. Необхідно розробити MVP (мінімально життєздатний продукт) за 3 місяці для впровадження на одному заводі. Навантаження на систему передбачуване і стабільне (робота в межах робочих змін заводу).

## Рішення:
Використовувати архітектуру "Модульний моноліт" із логічним розділенням на домени (Управління замовленнями, Склад, Виробництво, Якість) всередині одного розгорнутого додатку.

## Обґрунтування:
1.	Висока швидкість розробки: відсутність витрат часу на налаштування мережевої взаємодії між мікросервісами.
2.	Простота розгортання: потрібен лише один сервер для запуску всієї бізнес-логіки підприємства.
3.	Простота транзакцій: легко оновлювати склад і статуси цеху в одній транзакції БД.

## Альтернативи:
| Варіант | Чому відхилено |
| :--- | :--- |
| Мікросервісна архітектура | Занадто високий поріг входу та складність інфраструктури (DevOps, Kubernetes) для команди з 3 осіб. Overhead на мережеві запити. |
| Serverless | Не підходить для підприємства, де система повинна працювати у внутрішній мережі заводу (on-premise) без залежності від хмарних провайдерів. |

## Наслідки:
### Позитивні: Швидкий випуск продукту, легкий пошук багів (один лог-файл).
### Негативні: Якщо модуль "Звіти" сильно завантажить процесор, уповільниться робота всієї системи (наприклад, терміналів операторів).
